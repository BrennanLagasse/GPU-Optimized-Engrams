# Future Serving Optimization Report

Timestamp: 2026-04-21 17:10 EDT

Branch: `engrams-baseline-benchmarking`

## Executive Summary

This report is self-contained. It summarizes the larger serving ideas that were requested after the measured 40B static-batch optimization sweep, what was actually tried for each idea, and what remains blocked by runtime architecture.

The key conclusion is:

> The remaining high-upside serving ideas are not more static-batch tweaks. They require request-level KV-cache state. Once we have per-row or paged KV metadata, the most promising next measured implementations are continuous batching and prefill/decode disaggregation.

Current measured best GPU result:
- Hardware: `gpu003`, `8 x NVIDIA H200`
- Model preset: `target_40b_approx`
- Approximate model size: `39.98B` parameters
- Workload: 100 deterministic heterogeneous requests
- Input lengths: mean 128, max 1024
- Output lengths: mean 128, max 1024
- Total requested output tokens: 12,800
- Best measured deployable config: `optimized_cached + longest_input_first + compact + BATCH_SIZE=16`
- Best measured serving time: `163.306s`
- Speedup over `naive + random + static + BATCH_SIZE=8`: `29.90x`

New result in this report:
- Added a discrete-event serving architecture simulator for bursty/Poisson/closed arrivals.
- Calibrated the simulator so `static_b16_compact` equals the measured `163.306s` H200 repeat.
- Under bursty arrivals, the simulator estimates prefill/decode disaggregation with 4 prefill workers and 16 decode slots at `41.393s`, or `3.95x` faster than calibrated `static_b16_compact`.
- This is not a GPU benchmark. It is an implementation-priority signal showing that prefill/decode disaggregation is worth building once paged/per-row KV exists.

## Current Cache Blocker

The current optimized attention cache is batch-shaped:

```text
cache_k: [batch, cache_position, heads, head_dim]
cache_v: [batch, cache_position, heads, head_dim]
ptr_current_pos: one scalar shared by the whole batch
```

This design supports:
- static batches where all rows advance together
- physical compaction where all surviving rows still share the same position

It does not support:
- admitting a new request into one completed row while other rows have longer cached histories
- moving a request from a prefill worker to a decode worker without request-level cache metadata
- decode microbatch regrouping after prefill
- paged KV block reuse

Required refactor:
- per-request or per-row cache positions
- per-row attention length metadata
- request-to-cache-slot indirection
- either paged KV blocks or an equivalent cache-block allocator
- KV handoff metadata for prefill/decode disaggregation

## Idea-By-Idea Details

### 1. Continuous Batching

What was tried:
- Implemented idealized continuous refill simulation with configurable slot counts.
- Compared 8, 16, and 32 decode slots.
- Added mild TP-factor variants to estimate whether faster per-token execution changes the priority.

Result:
- Best simulated case: `continuous_slots32_tp1.15` at `17.824s`.
- Best non-TP continuous case: `continuous_slots32_tp1.00` at `18.576s`.
- This is the strongest simulated family, but it requires paged/per-row KV before it can be measured honestly on GPU.

### 4. Paged-Attention-Style Cache Layout

What was tried:
- Inspected the current cache shape and control flow.
- Documented why shared `ptr_current_pos` blocks refill, decode regrouping, and KV handoff.

Result:
- This is the enabling refactor for most future serving improvements.
- It was not implemented in this pass because it changes core attention semantics and needs dedicated parity tests.

### 5. Decode Microbatch Scheduler

What was tried:
- Implemented a simulator path that batches prefill separately from decode.
- Tested prefill batch sizes 8, 16, 32 and decode microbatch sizes 2, 4, 8.

Result:
- Simulated decode microbatch variants were not competitive with continuous batching or well-provisioned disaggregation.
- Still blocked for a real GPU benchmark because request KV state must be regrouped between prefill and decode batches.

### 6. Prefill/Decode Disaggregation

What was tried:
- Implemented separate prefill and decode worker pools in the simulator.
- Tested prefill worker counts 1, 2, 4.
- Tested decode slots 8, 16, 32.
- Tested `same_device` and `nvlink` KV handoff assumptions.

Result:
- Best simulated disaggregated case: `disagg_prefill4_decode16_same_device_tp1.00` at `41.393s`.
- Best NVLink handoff case: `disagg_prefill4_decode16_nvlink_tp1.00` at `41.777s`.
- This strongly supports prioritizing prefill/decode disaggregation after paged/per-row KV exists.

### 7. Tensor Parallelism

What was tried:
- Feasibility-probed against the current implementation.
- Simulated mild tensor-parallel speedup as `tp_factor=1.15` in the continuous batching family.

Result:
- `continuous_slots32_tp1.15` improved from `18.576s` to `17.824s`.
- Real tensor parallelism was not implemented because it requires sharded linear/attention layers, collectives, and new correctness tests.
- Current code uses contiguous layer placement, which is memory-oriented model parallelism, not tensor parallelism.

### 8. External Serving Baselines

What was tried:

```bash
python - <<'PY'
for mod in ['vllm','sglang']:
    try:
        __import__(mod)
        print(f'{mod}: installed')
    except Exception as exc:
        print(f'{mod}: not importable ({exc.__class__.__name__})')
PY
```

Result:
- `vllm`: not importable locally
- `sglang`: not importable locally

Interpretation:
- Installing either package would not by itself create a fair baseline because this repo has a custom Engram architecture.
- A fair external baseline requires either custom Engram integration or a clearly labeled non-Engram scheduler/runtime comparison.

### 9. Cost-Model-Driven Scheduler Search

What was tried:
- Implemented closed-batch cost-model search in `scripts/simulate_serving_strategies.py`.
- Implemented bursty-arrival architecture simulation in `scripts/simulate_serving_architectures.py`.

Result:
- Closed-batch model suggested modest gains from ideal continuous refill, around `156.995s` vs measured `163.306s`.
- Bursty-arrival simulation showed much larger gains because it captures stage overlap and head-of-line blocking.
- This explains why prefill/decode disaggregation looked less important in the earlier closed-batch framing but much more important in a serving-style arrival process.

## Recommendation

Do not spend more time on static-batch scheduler tweaks unless more variance estimates are required.

Recommended implementation order:

1. Implement per-row or paged KV-cache metadata.
2. Add correctness tests proving cache-equivalent generation for heterogeneous row lengths.
3. Implement true continuous batching with refill-on-completion.
4. Add prefill/decode disaggregation with KV handoff.
5. Benchmark under bursty and Poisson arrivals, reporting makespan, TTFT, TPOT, mean latency, P95 latency, and GPU utilization.
6. Consider tensor parallelism after the serving runtime can keep GPUs busy with continuous batches.

If project scope cannot support the KV-cache refactor, the current best measured result should stand:
- `MODEL_IMPL=optimized_cached`
- `POLICY=longest_input_first`
- `DECODE_MODE=compact`
- `BATCH_SIZE=16`
- Serving time: `163.306s`
- Speedup over naive random static: `29.90x`
